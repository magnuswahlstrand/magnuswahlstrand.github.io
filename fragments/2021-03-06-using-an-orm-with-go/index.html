Using an ORM with Go

A while back @brandur’s article Using Atomic Transactions, I realised that I have actually never used an ORM (wiki).

When I started web programming in Go, I quickly adopted the community notion (from forums, and presentations) that using large frameworks is frowned upon. Was I wrong?

I think the aversion to ORM-like frameworks might be caused by Go’s design principle to show the underlying complexity of certain operations.

For example:

// Go
for _, u := range users {
    if u.Type != "admin" {
        continue
    }
    adminUsers = append(adminUsers, u)
}

is preferred over something like this

// Java
CollectionUtils.filter(result, o -> ((User) o).getType() == 'admin');

For web development, many people seem to prefer using the standard library, or lightweight packages like jmoiron/sqlx (for SQL) or go-chi/chi (for routing).

I have used these lightweight tools extensively, and they are great! It has forced me to learn a lot about databases, SQL, and building REST APIs for production environments.

But (sometimes ðŸ™‚) I think, why can’t Go be like Ruby?

Most hobby projects fall into these categories, that make them good candidates for using an ORM:

  • Have very few users (usually just me), meaning performance requirement is low
  • Require no migrations
  • Require no advanced queries
  • Are built to validate or test an idea

There are downsides to using something more complex like an ORM, but after having worked professionally with Go and web development for more than 3 years. I be ready to handle them.
